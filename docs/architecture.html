ET AI Hackathon 2026 · Problem Statement 8

Plant Brain

A unified industrial knowledge platform. It ingests heterogeneous plant documents into a single knowledge graph, answers questions with citations that resolve to a page or an audio timestamp, watches equipment history for failure patterns, captures retiring operators' spoken knowledge in any language, and maps quality findings back to the standard clause that governs them. Fully local — no cloud API at any point.

System architecture

Four agents operate on a blackboard: no agent ever calls another. They read and write shared graph state, which means a new agent can be added without modifying a single existing one. Data flows in one direction — generated corpus, into the graph, out through the query agents.

The critical structural decision is the split ingestion path. Clean tabular data (equipment registers, work orders, KPIs) loads directly to typed nodes with no LLM involvement whatsoever; only narrative documents go through chunking and embedding. This drops the 7B model's workload from roughly 500 chunks to about 40, cutting extraction from ~50 minutes to under 5, and removes hallucination risk from the cleanest data in the system.

Local model allocation

The published spec assumed an 8 GB GPU; the build machine has 6 GB, so allocation was re-benchmarked rather than assumed. The 7B model splits 82% GPU / 18% CPU and sustains 22–28 tokens/sec, which was measured as stable and therefore kept. A 4B alternative fit entirely in VRAM but showed severe throughput variance (71 tok/s, then 4.9) and was demoted to emergency fallback.

Knowledge graph schema

One Neo4j instance holds everything, but in two deliberately different forms. Document content carries 768-dimension embeddings for semantic retrieval. Facts and measurements are typed nodes queried by Cypher and computed over — never embedded as text, because a vector search cannot do arithmetic or detect a trend.

The one schema rule that cannot be broken

:TacitKnowledge and :Procedure are separate labels and must stay separate. Spoken operator experience is one person's unverified practice; a procedure is approved policy. Merging them would be unsafe in a plant context — and keeping them apart is the only reason divergence detection is possible at all.

The four agents

P1 · Ingest & Build

Takes the corpus directory, produces a populated graph, along three paths. Structured: spreadsheets load straight to typed nodes — no LLM touches already-typed data. Narrative: SOP PDFs chunk on heading and numbered-step boundaries at ~600 tokens with 15% overlap; a regex pass extracts equipment tags first, and only then does the LLM bind relations between entities that are already known, against a closed vocabulary. Entity resolution: every tag is canonicalised and merged to one node carrying an alias list.

P2 · Voice Capture awaiting clips

Audio in any language becomes typed :TacitKnowledge claims. Whisper runs with task="translate", so any source language reaches English in a single pass with no separate translation model, preserving per-segment timestamps and the detected language. Critically, ASR output is never trusted for equipment tags — spoken "P-101A" transcribes as "pee one oh one ay", so the transcript is normalised for spelled-out digits and passed through the tag regex, and the LLM is given only the verified candidates.

P3 · Ask

A question becomes a cited answer with a computed confidence score. Detailed in §04.

P4 · Watch

Runs on a 30-minute timer. Every detection is arithmetic and Cypher; the LLM only writes the narrative afterwards. That separation is what makes findings defensible — the numbers are never generated. Six detectors currently fire:

Procedure-vs-practice divergence — the differentiating finding

The written SOP and what operators actually do drift apart, and nobody notices. Detection requires a same-topic gate before comparison: a procedure and a spoken claim must apply to the same equipment and reference the same maintenance activity before their stated frequencies are compared at all. Comparing by embedding similarity alone would fire constantly on a small corpus.

One subtlety the implementation had to handle: operators typically cite the official interval while rejecting it — "the book says monthly, we do it fortnightly." A naive set intersection sees "monthly" in both texts and concludes agreement. The detector instead isolates the frequency the operator states that the procedure does not, which is the actual deviating practice.

04

Retrieval router

Both retrieval types are required because they fail on opposite question shapes. Keyword search catches exact tags and standard codes that embeddings miss; dense retrieval catches paraphrase — "won't build pressure" matching "insufficient discharge head". Neither can answer a question where the answer document shares no words with the question at all: a report on P-101C's failure never contains the string "P-101A", so no text search reaches it. Only walking equipment → same class → other equipment does.

Therefore both run in parallel every time, and an intent classifier sets the blend ratio rather than branching exclusively. Misclassifying an exclusive branch returns nothing; a wrong blend still returns something useful.

Confidence is computed, never self-reported

A model asked to rate its own certainty produces a number with no grounding. Here the score is arithmetic over observable signals: normalised top retrieval score, the fraction of answer sentences that actually carry a citation, and the overlap between the keyword and dense result sets. Below threshold the system abstains — in a safety context, "not present in the corpus" is the correct answer, and a confident wrong answer about plant equipment is worse than no answer.

05

QMS integration

Quality management is implemented as a specialization of the same architecture, not a second system: same graph, same agents, same retrieval logic. Only the ingested standards pack and the KPI definitions differ.

Clause-aware chunking

The quality standard splits on clause numbers, not fixed token windows. A clause is the natural retrieval unit in a compliance context — splitting one mid-sentence produces answers that cite half a requirement, which is worse than useless in an audit.

Numbers are never embedded

This is where pure-RAG architectures typically fail. Process capability and defect rates load into typed :QualityKPI rows that an agent queries and computes over directly. Embedding "Cpk = 1.21" as a sentence and hoping a vector search retrieves it correctly cannot reliably support arithmetic, threshold tests, or trend detection. The thresholds themselves (Cpk minimum 1.33; PPM rising three consecutive months) come from configuration, so every computed figure is reproducible and carries the exact query that produced it.

Compliance mapping

Non-conformance records link equipment to the clause they violate, which is what lets a question like "why is PPM rising on the feed transfer process" traverse from a number, to the process, to the equipment, to the open NCR, to the governing clause 8.5.1 — and then to the monsoon strainer story that explains it. No vector search reconstructs that chain.

Industry-agnostic by configuration

The pipeline is fixed; the standards pack swaps. A configuration layer defines which base standard applies, its clause taxonomy, and the industry's KPI definitions — defect rate for automotive, batch yield for pharma, lot traceability for food. Graph schema and agent logic stay identical. That is what makes this one architecture rather than one build per industry.

06

Feature inventory

Twenty-six implemented capabilities, grouped by the part of the system they belong to. The indented note under each states why it exists — in most cases the design choice was forced by a specific failure it prevents.

Knowledge construction

A1 Deterministic corpus generator

Every document, register and record is produced by a seeded script rather than hand-authored, and the script asserts its own output — the planted failure patterns are verified on every run.

Why: when the schema changes you regenerate rather than rewrite, and a corpus that verifies itself cannot silently lose the patterns the entire demo depends on.

A2 Entity resolution across four spellings

The same pump appears as P-101A in work orders, Pump 101A in procedures, P101-A in an inspection record, and p-101a in a voice transcript. All four canonicalise to one node carrying an alias list.

Why: this is the single point of failure in the whole system. Three nodes for one pump means every multi-hop query silently returns nothing — not an error, just emptiness. It is verified by hand before anything downstream is trusted.

A3 Structured-first ingestion

Spreadsheet data becomes typed nodes with no LLM in the path; only narrative documents are chunked and embedded.

Why: running already-typed data through an extraction model introduces hallucination into the cleanest source available, and it cuts the model's workload by more than 90%.

A4 Regex-before-LLM relation binding

In narrative documents a regex pass finds equipment tags, dates and measurements first; the LLM then only binds relations between entities already found, against a closed relationship vocabulary.

Why: a 7B model is weak at open-ended entity extraction but adequate at choosing among known candidates. This plays to that boundary instead of across it.

A5 Failure-code correction

A deliberately mistyped ISO 14224 code in the corpus is mapped back to the closed vocabulary on load, with the raw value preserved on the node.

Why: real maintenance records contain typos; a closed vocabulary is what makes the reliability analytics work at all, so entries must be repaired rather than dropped.

A6 Clause-aware chunking

The quality standard splits on clause boundaries rather than token counts.

Why: a compliance answer that cites half a requirement is worse than no answer.

A7 Structured quality store — numbers never embedded

Cpk and PPM values are typed rows queried and computed over, not text in a vector index.

Why: embeddings cannot do arithmetic or trend detection. This is the specific thing pure-RAG quality systems get wrong.

Answering

B1 Hybrid retrieval with RRF fusion

Keyword and dense retrieval run in parallel on every question and fuse by reciprocal rank.

Why: they fail on opposite question shapes — exact tags versus paraphrase. Running only one guarantees a class of question returns nothing.

B2 Graph expansion in a single round trip

From the retrieved seeds the query traverses two hops — equipment → work orders → failure modes → sibling equipment → operator knowledge — inside the same Cypher statement.

Why: it answers questions where the answer document shares no vocabulary with the question, and doing it in one statement keeps latency flat rather than multiplying round trips.

B3 Multi-form anchoring

A question anchors on an explicit tag, a class or area phrase ("the control valves in the process area"), a process name, or a clause number.

Why: people rarely quote asset tags. Without phrase anchoring, questions that named a class instead of a tag retrieved procedure text only and wrongly abstained — a failure found in testing and fixed.

B4 Blend ratio, never an exclusive branch

Intent classification adjusts how much text versus graph evidence is assembled; it never switches one off.

Why: misclassifying an exclusive branch returns nothing. A wrong blend still returns something useful.

B5 Symptom-aware evidence ordering

Question wording maps to ISO failure modes — "vibration" to VIB — and matching history rows sort to the top of the evidence block.

Why: a genuine failure found in testing. The relevant vibration record sat seventh of seven in a table, the model overlooked it, and the system abstained on a question it could answer.

B6 Analytics leg for numeric questions

Capability and trend questions attach computed KPI evidence — the series, the threshold test, and an explicit verdict flag — assembled ahead of document text.

Why: two further failures found in testing. Buried mid-prompt the model hedged around its own computed answer; and when the block said only "PRC-301" the model could not connect it to a question naming "the feed flow control process", so it correctly refused. Naming the process in the evidence fixed it.

B7 Computed confidence with abstention

Score is arithmetic over retrieval strength, citation density and result-set agreement. Below threshold the system declines to answer.

Why: self-reported confidence is ungrounded. In a plant context a confident wrong answer is more dangerous than no answer.

B8 Resolvable citations

Every claim carries a marker resolving to a document page, an audio timestamp, or the exact query run against the structured store. Clicking one opens the PDF at that page or plays the clip at that second.

Why: in a compliance context an uncited reference is worse than no answer, and a citation you cannot open is just a claim.

B9 Streaming responses

Answers stream token by token over server-sent events; models pre-warm at startup.

Why: first tokens now appear in about two seconds instead of a blank ten-to-twenty second wait — and pre-warming removes the cold-start stall entirely.

B10 Show-the-query toggle

The Cypher behind any answer is inspectable from the answer itself.

Why: makes the system auditable rather than magic — the difference between a demo and a tool an engineer would trust.

Proactive intelligence

C1 Reliability detection — overdue, chronic, sibling exposure

MTBF and risk ratio from failure intervals; repeat-mode counting; same-class propagation on failure.

Why: all three are arithmetic, so a finding can be defended line by line. Sibling exposure in particular turns one pump's failure into a fleet-wide preventive action.

C2 Procedure-vs-practice divergence needs clips

Detects where the written SOP and actual operator practice have drifted apart, gated on shared equipment and shared activity before any frequency comparison.

Why: this is the finding no document system surfaces, because it requires holding unverified spoken practice and approved policy as separate things and then comparing them. Proven against synthetic claims; awaiting the real recordings.

C3 Quality detection — capability and defect trend

Cpk below the configured minimum, and defect rate rising for three consecutive months.

Why: computed from the structured store rather than retrieved, so the verdict is a calculation, not a paraphrase.

C4 Business impact in hours

Each alert states the avoidable downtime, derived from the historical downtime of that failure mode on that equipment class.

Why: turns a technical finding into a number a plant manager acts on. Narratives are explicitly constrained to facts in the data after the model invented a "$10,000 per hour" figure during testing.

C5 Evidence chains on every alert

Alerts link back to the specific work orders, chunks and claims that produced them.

Why: a finding you cannot trace is a finding nobody will act on.

C6 Compliance mapping

Findings map to the governing clause through the non-conformance register.

Why: an auditor's first question is which requirement was breached, and that traversal is graph-shaped.

Interface and evidence of correctness

D1 Knowledge graph explorer

Interactive force-directed view with an architecture-level overview and a per-equipment focus mode. Written without any external library.

Why: a CDN-hosted graph library fails on an air-gapped demo laptop. The focus view also doubles as visual proof that entity resolution worked.

D2 Entity resolution panel

Shows every merged spelling and what it resolved to.

Why: makes the least visible and most important part of the pipeline demonstrable in one glance.

D3 Time-to-answer stopwatch and detected-language badge

Live timing on every question; source language shown on spoken claims.

Why: both are explicit judging criteria, and both expose something already computed rather than adding a failure mode.

D4 Evaluation harness with retrieval ablation

Twenty-one questions scored across four retrieval configurations, producing a chart.

Why: it converts "our retrieval design is good" into a measurement, and the ablation isolates exactly what the graph contributes.

D5 Configurable standards pack

Standard identity, clause taxonomy and KPI thresholds live in configuration.

Why: it is the difference between one architecture and one build per industry.

The ablation makes the architectural argument precisely: removing graph traversal fails exactly the questions whose answers share no vocabulary with the question. That is the class of question a vector database alone cannot serve, and it is the reason the knowledge graph exists rather than a document index.

Honest note on the single-leg results. Dense-only and keyword-only score higher than they would in a pure-RAG system because graph anchoring still carries them — the text legs are crippled but the equipment traversal is not. The clean, defensible finding is the no-graph row.
08

Scale & status

Why four agents

The production design is twelve, across four tiers — ingestion, query, intelligence, delivery. The prototype implements four; the remaining eight are email triage, Cypher generation, root cause analysis, compliance auditing, lessons-learned clustering, multi-channel notification, drawing parsing and OCR.

Because the architecture is a blackboard — agents read and write shared graph state and never call each other — adding the missing eight requires changing zero existing agents. Knowing precisely what was deferred, and why, is engineering judgment rather than omission.

Deliberately out of scope

Drawing and P&ID parsing, OCR, and email ingestion were cut on purpose. Each is a full day of work carrying real risk, and none adds a capability the platform does not already demonstrate through another door. The GPU headroom they would have consumed was spent on stability instead.
